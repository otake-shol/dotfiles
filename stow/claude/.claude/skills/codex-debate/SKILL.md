# Codex 自動議論（敵対的相互検証）

自分（Claude）の結論と Codex（ChatGPT）の結論を敵対的に突き合わせ、**収束点（共同声明）と残った対立点（crux）を分離して**ユーザーに報告する。目的は「勝つこと」ではなく、ユーザーに最も信頼できる最終推奨を返すこと。

## 前提チェック

1. `codex --version` で CLI の存在を確認（なければユーザーに報告して終了）
2. **自分の独立判断を先に固める**——Codex の意見を読む・送る前に、同じ材料から自力で結論と確信度を出す（アンカリング回避。これを飛ばすと議論が追認になる）
3. 機密情報を含むディレクトリで実行する場合、その内容が Codex（OpenAI）にも読まれることをユーザーが了解しているか初回に確認する

## 実行手順（1ラウンドごと）

1. プロンプトを scratchpad に `.md` で書く（クォート地獄回避のため stdin 渡しにする）
2. 実行:
   ```bash
   cd <対象ディレクトリ> && codex exec --sandbox read-only - < <scratchpad>/roundN-to-codex.md
   ```
   - timeout は 600000（Codex がファイルを読むラウンドは数分かかる）
   - 対象ファイルを読ませたい場合は該当ディレクトリで実行し、ファイルパス一覧をプロンプトに明記
   - **必ず `--sandbox read-only`**。ファイル編集は絶対にさせない（プロンプト内でも「読み取りのみ」と明記）
3. 応答を読み、譲歩・更新・矛盾を分析してから次ラウンドを設計する

## ラウンド設計（基本3ラウンド、上限5）

- **R1 反論状**: ①相手の主張を引用 ②自分の結論と確信度 ③番号付き反論（相手の前提・条件設定の急所を突く。条件付き結論なら条件の成立確率を掛け戻す）④相手への質問（重み付け・確率見積もりの根拠を問う）⑤自分の結論への最強の自己反論も開示する（相手が使ってよいと明記——誠実さが収束速度を上げる）
- **R2 レッドチーム**: 収束したら「合意結論を全力で崩せ」を依頼。**2つのAIが共有しがちなバイアス**（現状維持バイアス・可逆性の過大評価・提供資料の自己記述への過剰信頼・ユーザーの語りへの迎合）を攻撃候補として明示的に列挙して渡す
- **R3 矛盾解消・最終合意**: 相手の回答内の論理矛盾（前ラウンドで自分が置いたゲートを踏み抜く等）を指摘し、修正提案→共同声明（3行以内）＋双方の最終確信度で締める
- 停止条件: 収束宣言が出る／2ラウンド連続で新しい論点が出ない／5ラウンド到達（その場合は残cruxを明記して打ち切り）

## プロンプトの型（毎ラウンド共通で入れるもの）

- 「これはAI同士の敵対的検証議論。忖度せず、間違っていると思う点は明確に反論せよ。ただし収束可能な点は収束させよ」
- **出力形式の厳守指定**（例: 各論点に「同意/部分同意/反論」＋理由3行以内、更新後の推奨と確信度0〜100%、残cruxの列挙 or「収束」明記）。形式がないと収束判定ができない
- ファイル編集禁止の明記

## 分析時の規律

- 相手の全面同意に安易に乗らない——同意が早すぎたら R2 で自分側の結論も攻撃させてバランスを取る
- 「生き残った攻撃」が結論の**方向**に向いているか、**条件の未確定**に向いているかを区別する（後者なら結論は維持し、条件リストを成果物にする）
- 相手が挙げた外部出典（リンク・統計）は鵜呑みにせず、判断の重みは「議論の論理」に置く

## 報告の型（最終ターンでユーザーに返すもの）

1. **最終結論（共同声明）**と双方の確信度——冒頭に置く
2. ラウンド経過表（各ラウンドで誰が何を譲歩・更新したか）
3. レッドチームで生き残った攻撃と、それによる結論の修正
4. 残crux（事実確認でしか閉じない項目）
5. コスト目安の一言（Codex 側は1ラウンド10万〜25万トークン級になることがある）

## 派生パターン

- **結論の頑健性テストのみ**: 自分の結論が既にあり相手の意見がない場合、R2（レッドチーム）から開始してよい
- **合意済み文書のレビュー**: 対象ディレクトリの文書群を読ませ、「この計画の前提の穴を探せ」型で1ラウンド使い切りも可
- ユーザーが議論結果の永続化（Vault・リポジトリへの反映）を求めたら、議論ログの保存版ノート＋既存文書への要点追記の二層構成を提案する（scratchpad はセッション終了で消える）
